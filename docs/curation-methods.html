Curation Methods

Inclusion criteria, data access definitions, metadata standards, and limitations

Author

Dahham Alsoud

This page documents the curation rules used to build and maintain the IBD Omics Explorer dataset registry. Understanding these rules is essential for interpreting the annotations and deciding whether a listed dataset is appropriate for a specific analysis.

Purpose and Scope

The IBD Omics Explorer is designed to map and organize publicly available IBD omics datasets across all major omics layers, with the goal of supporting dataset discovery. It is not a systematic review, meta-analysis, or quality-assessment resource. Inclusion in this registry does not imply that a dataset is high quality, fully reusable, or suitable for any particular analysis.

The registry aims to be inclusive and expandable. The current dataset is a starting point, not a comprehensive catalog. Many datasets that qualify for inclusion may not yet be listed.

Current Registry State

The current dataset registry contains 116 primary dataset entries plus 10 integrated resources tracked separately. The newer systematic-discovery rows are intentionally marked as source_checked rather than fully verified until their publication links, sample counts, and cohort labels are reviewed row by row. This means the website is most reliable as a discovery map and triage tool, not as a final source of study-level metadata.

Before using an entry for analysis, verify the source repository record directly, including disease groups, sample counts, specimen type, assay platform, access conditions, and publication links. Entries with Unclear fields should be treated as leads for manual curation rather than confirmed facts.

Retired Primary Dataset IDs

Some DSxxx identifiers are intentionally retired when a broad portal, consortium, or integrated resource is moved out of the primary accession-level dataset table. Retired identifiers are not reused. This keeps older references interpretable while preventing resource-level rows from being counted as single primary datasets.

Retired identifiers are tracked in data/retired_dataset_ids.csv.

Priority Verification Queue

The following entry types should be checked first because they carry the highest risk of misleading users:

Priority Entry pattern Why it matters
1 Entries with dataset_url = Unclear Users cannot verify the source record directly from the site
2 Entries with publication_title = Unclear and publication_year = Unclear Study provenance is incomplete
3 Aggregated portals or multi-accession resources These can mix studies with different assays, cohorts, and access rules
4 Entries where sample type, platform, or disease groups are partly inferred These fields determine whether datasets can be pooled or compared
5 Controlled-access genetics or multi-omics resources Public indexing does not mean the underlying individual-level files are downloadable

Concrete current examples include GSE accessions with unknown publication metadata and treatment-trial rows where sample counts or publication linkage still need row-level checking. Integrated databases, atlases, and broad genetics/platform entries are tracked separately on the Integrated Resources page.

Inclusion Criteria

A dataset is eligible for inclusion if it meets at least one of the following criteria:

  1. Human subjects with inflammatory bowel disease (IBD), Crohn’s disease (CD), ulcerative colitis (UC), or IBD-unclassified (IBD-U)
  2. Non-IBD control samples collected in the context of an IBD study
  3. Human intestinal, colonic, ileal, blood, stool, organoid, or microbiome samples analyzed with omics methods in an IBD research context
  4. A publicly registered accession, portal entry, or publicly accessible summary statistics
  5. A curated database specifically aggregating IBD omics studies (e.g., IBDTransDB)
  6. A controlled-access resource that is publicly indexed (e.g., dbGaP) and thus searchable and describable

Exclusion or Deferred Categories

The following categories are excluded or deferred pending additional review:

Category Handling
Animal-only datasets (mouse, rat models) Excluded unless they directly describe human companion datasets
Purely mechanistic cell-line datasets Excluded unless they include primary patient-derived material
Datasets without any accessible metadata Deferred until metadata can be located
Duplicate submissions Retained only if they add useful metadata, processed files, or additional accession cross-links
Publication-only datasets Included with Publication only or Unclear access status and a note
Unpublished preprint-only datasets May be included with explicit annotation; treat with caution

Public Data Access Labels

The registry uses the following standardized labels for the data_access_status field:

Label Meaning
Public raw data Raw data files (FASTQ, CEL, IDAT, mzML, mzIdentML, mzTab, count matrices, BAM, or equivalent) are publicly downloadable without application
Public processed data Processed expression matrix, abundance table, or summary statistics are publicly available, but raw data may not be
Public portal Data are accessible and explorable through a public portal or database interface, with or without direct file download
Controlled access Dataset is publicly indexed and described, but access to data files requires an application, institutional approval, or data use agreement
Publication only Dataset is described in a publication, but reusable data files are not clearly available in a repository
Unclear A repository record exists but data availability, metadata, or access conditions could not be confidently determined from public information

Multiple labels may apply to a single dataset (e.g., Public processed data; Controlled access (some components)).

Curation Status Labels

Each entry carries a curation_status label indicating the degree of metadata verification:

Label Meaning
Needs verification Metadata or access status should be manually checked before relying on the annotation
Partially verified One or more source, metadata, access, or citation checks have been completed, but the row is not analysis-ready
Verified Metadata and access links have been manually checked against the source repository record
Deprecated/limited Dataset exists but has limited reuse value, incomplete public access, or substantial data quality issues

Most entries currently require additional verification. Users are encouraged to verify directly against source repositories before initiating analysis.

Verification Workflow

The verification workflow is deliberately conservative:

  1. New rows start as pending_review.
  2. A row moves to source_checked only after the source URL opens and matches the intended accession or resource.
  3. A row moves to metadata_checked only after key study metadata are checked against a source record.
  4. A row moves to publication_checked only after citation fields are checked against a publication, DOI, PubMed, or repository-linked citation.
  5. A row becomes needs_revision if source evidence contradicts the current registry row.
  6. A row should only become analysis_ready_candidate after human review of source, metadata, publication, access status, and the intended analysis context.

Verification batches cover selected high-value/high-risk entries across multi-omics, single-cell, proteomics, metabolomics, methylation, transcriptomics, and controlled-access genetics resources. Atlases, portals, curated databases, and consortium resources are tracked on the Integrated Resources page because they are not single accession-level datasets. Systematic-discovery additions are retained as source_checked until they receive deeper row-level verification.

Cohort Families and Overlap

Some registry rows describe different accessions, assays, timepoints, or derived resources from the same underlying study population. These rows are useful as separate dataset records, but their subject or sample counts should not automatically be summed.

When available, cohort-family review fields identify related rows, paired modality derivatives, and rows that should be treated as components of a shared study family rather than independent cohorts. Rows marked as uncertain should be interpreted conservatively until source-level sample IDs or study documentation are reviewed.

For de-duplicated cohort counts, use the cohort-family audit files rather than summing the main dataset table directly.

Therapeutic-family reviews are included when available, but rows requiring source-level sample-ID review should be treated as provisional for cohort counting.

The primary dataset CSV now includes appended cohort-family fields (study_family, component_of_dataset_id, count_as_independent_cohort, and overlap_notes). These fields are intentionally conservative and currently appear in the Dataset Explorer detail view only.

All 116 primary dataset rows now have a study_family label, representing 95 distinct study or source families. count_as_independent_cohort = FALSE means the row should not be counted as a separate cohort because it is a known component, subset, or overlapping accession for another row. Unclear means the row is related to another study family, but public metadata are not sufficient to reconcile donors or samples confidently. Users should treat Unclear rows conservatively and avoid summing them without source-level review.

Users can search by study, cohort, or trial names such as PROTECT, RISK, IBD200, ACT1, UNITI, UNIFI, GEMINI, BERGAMOT, scIBD, or SP140. These labels are intended for discoverability and cohort accounting; they do not replace direct verification of the source repository before analysis. Current countability values are 93 TRUE, 9 FALSE, and 14 Unclear.

Metadata Caveats

Users of this registry should be aware of the following general caveats:

Sample size. Sample size fields may refer to subjects, unique samples, biopsies, cells, sequencing libraries, or files, depending on the study. These are not always interchangeable. Actual analyzable sample sizes per disease group should be verified from the publication or repository metadata.

Disease definitions. IBD, CD, UC, and IBD-U definitions vary across studies. Some studies use clinician-assigned diagnoses; others use ICD codes or self-report. Disease activity definitions (active vs. remission) and scoring systems differ. Cross-study comparisons of disease groups require attention to diagnostic criteria.

Treatment exposure. Treatment status at the time of sample collection is often incompletely annotated in public deposits. Even where annotated, the specific agents, dosing, duration, and response status may not be consistently available.

Pediatric vs. adult. Age group classification may not always be explicitly labeled. Unlabeled entries should not be assumed to be adult. Biological differences between pediatric and adult IBD are well established; pooling across age groups is generally not recommended without justification.

Tissue site. Tissue site annotation may be incomplete, inconsistent across studies, or ambiguous (e.g., “colon” without specifying anatomical segment). Cross-study comparisons should account for tissue site differences.

Platform effects. Different omics platforms (microarray vs. RNA-seq; different sequencing depths; different array generations) introduce systematic differences that must be accounted for in cross-study analyses.

Batch effects. Omics data are subject to batch effects from sample processing, sequencing run, laboratory, and reagent lot variation. Batch correction is typically required for cross-study comparisons.

Controlled-access data. Controlled-access resources are described for completeness but require institutional approval before any data can be downloaded or analyzed. Approval processes vary and can take weeks to months.

Unclear fields. Fields marked “Unclear” indicate that the information could not be confidently determined from publicly available records at the time of curation. These should be verified before use.

How to Contribute or Correct

This registry is designed to expand continuously. If you identify:

  • A dataset that should be included
  • A metadata error or outdated access status
  • A deprecated or moved accession
  • A verified entry that should be updated from Needs verification to Verified

Please open an issue or pull request on the source repository.

Data Sources for Curation

Metadata in this registry should be checked against:

  • Public repository records, including GEO, SRA, PRIDE, MassIVE, Metabolomics Workbench, dbGaP, GWAS Catalog, CELLxGENE, and Human Cell Atlas
  • Consortium and project portals, including IBDMDB, IIBDGC, Open Targets, FinnGen, and UK Biobank
  • Primary publications and supplementary tables linked from repository records
  • Dataset-specific README files, sample metadata tables, and controlled-access study descriptions

The initial entries should not be promoted to Verified until a human curator has checked the repository record and primary publication.

This registry is a curated research resource and is provided for dataset discovery purposes only. It does not constitute clinical guidance, a systematic review, or a quality certification of any dataset. All metadata should be independently verified before use in any analysis.
